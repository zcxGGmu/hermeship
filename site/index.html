Hermeship

面向 Hermes 的 daemon-first 事件通知路由器。它接收 Hermes gateway hooks、可选 observer plugin、CLI 和本地 deterministic source 事件， 经过 typed envelope、privacy scrub、route、render 和 sink delivery，把结果送进通知渠道而不是把噪音留在 agent 上下文里。

工作流路径

ingest → normalize → scrub → route → render → deliver

每一跳都独立、可解释、可失败。Hermeship 先把事件变成 typed envelope，再决定投递到哪里，以及是否需要裁剪敏感内容。

  1. 01

    Ingest

    接入 gateway hooks、CLI、observer plugin 和本地 source 命令。

  2. 02

    Normalize

    统一 event envelope、body 和 metadata，避免上游格式飘移。

  3. 03

    Scrub

    先做隐私清洗，再进入路由与渲染，避免原始噪音外泄。

  4. 04

    Route

    按事件、平台、项目和 route filter 选择目标 sink。

  5. 05

    Render

    把事件变成适合通知渠道的 compact、inline 或 alert 形式。

  6. 06

    Deliver

    交给 sink 执行投递，失败也要回传明确的边界信号。

运行边界与验证

哪些能说，哪些必须说清楚不能说成已完成

Hermeship 的价值在于明确边界。这个页面不把后续范围写成现成能力，也不把本地 preflight 讲成线上 live pass。

  • Hermeship 不修改 Hermes core，也不把通知消息写回 Hermes 对话上下文。
  • Observer plugin 需要 operator 显式安装并手动启用。
  • Slack sink 不在默认范围，页面不能暗示已经支持。
  • GitHub API polling 仍是后续范围，页面不能暗示已上线。
  • Real Discord/Hermes live verification pass 未获得，页面不能写成已完成。
  • release preflight 只证明记录字段存在，不证明线上环境已经通过。
  • 页面中不出现 raw prompt、conversation history、request/response body、secret 或 session_key 的外发暗示。
