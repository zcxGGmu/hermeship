Method

Observe · Scrub · Deliver

首页保留工程细节，但对外先给出更容易记住的三步方法论。

01

Observe

从 Hermes hook、CLI、本地 source 和可选 observer plugin 捕获事件，不把 agent 上下文当消息队列。

02

Scrub

先做 typed envelope 与 privacy scrub，只保留路由、渲染和审计需要的安全字段。

03

Deliver

按 route filter 选择目标 sink，输出 compact、inline 或 alert 形式，并保留清晰失败边界。

工作流路径

ingest → normalize → scrub → route → render → deliver

每一跳都独立、可解释、可失败。Hermeship 先把事件变成 typed envelope，再决定投递到哪里，以及是否需要裁剪敏感内容。

  1. 01

    Ingest

    接入 gateway hooks、CLI、observer plugin 和本地 source 命令。

  2. 02

    Normalize

    统一 event envelope、body 和 metadata，避免上游格式飘移。

  3. 03

    Scrub

    先做隐私清洗，再进入路由与渲染，避免原始噪音外泄。

  4. 04

    Route

    按事件、平台、项目和 route filter 选择目标 sink。

  5. 05

    Render

    把事件变成适合通知渠道的 compact、inline 或 alert 形式。

  6. 06

    Deliver

    交给 sink 执行投递，失败也要回传明确的边界信号。

Evidence & Boundaries

信任来自证据，而不是把后续范围写成现成能力

Hermeship 的价值在于明确边界。页面中的每个状态都要能回到 README、spec、development-status 或 runbook，而不是只靠营销文案支撑。

Local proof deterministic smoke

Quickstart 证明本地命令和 event pipeline 能跑通，不证明真实外部系统。

Release proof preflight records

preflight 证明记录字段存在，不等于 Discord/Hermes live pass。

Operator proof manual observer enablement

observer plugin 由 operator 显式安装和启用，不自动改 Hermes 配置。

  • Hermeship 不修改 Hermes core，也不把通知消息写回 Hermes 对话上下文。
  • Observer plugin 需要 operator 显式安装并手动启用。
  • Slack sink 不在默认范围，页面不能暗示已经支持。
  • GitHub API polling 仍是后续范围，页面不能暗示已上线。
  • Real Discord/Hermes live verification pass 未获得，页面不能写成已完成。
  • release preflight 只证明记录字段存在，不证明线上环境已经通过。
  • 页面中不出现 raw prompt、conversation history、request/response body、secret 或 session_key 的外发暗示。

如果某个叙述不能被 README、spec 或 development-status 直接支撑，就不要放到首页里。
